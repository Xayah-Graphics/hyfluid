\section{Dataset Preparation}\label{sec:dataset-preparation}

\subsection{ScalarReal Dataset}\label{subsec:scalarreal-dataset}

ScalarReal is this project's local five-video subset of ScalarFlow, introduced by \citet{eckert2019scalarflow} for real-world scalar transport. The public release contains 104 reconstructions, each with 150 frames of 3D density, 3D velocity, rendered images, and input images~\citep{scalarflowData2019}. This subset contains five camera videos and \texttt{info.json}; tables follow JSON field order.

Use labels: \emph{train} is used by Python density training; \emph{test} is held out; \emph{render} builds the synthetic render path; \emph{no} is unused there.

\subsection{Device Dataset Layout}\label{subsec:device-dataset-layout}

After initialization, each host frame set \(H_f\) has a device record \(D_f\) at the same index. The record is not a packed struct on the GPU; it is four separately allocated device buffers plus frame-set shape metadata kept on the host. Let
\[
    N_f=V_fT_f
\]
be the dense frame count for a frame set with \(V_f=\texttt{view\_count}\) and \(T_f=\texttt{time\_count}\). The loaded frames must cover every view-time pair.

The sampler and evaluator first resolve a frame index from the dense grid:
\[
    k=\texttt{frame\_indices}[vT_f+t].
\]
They then read \(\texttt{camera}+12k\), \(\texttt{intrinsics}+4k\), and \(\texttt{pixels}+4kWH\). In the CUDA kernels, the 12 camera floats become
\[
    x_c=\texttt{camera}[0{:}3],\quad
    y_c=\texttt{camera}[3{:}6],\quad
    z_c=\texttt{camera}[6{:}9],\quad
    o_c=\texttt{camera}[9{:}12],
\]
where \(x_c,y_c,z_c\) are the ray basis in field space and \(o_c\) is the ray origin. All sampled positions therefore live directly in the unit AABB training domain \([0,1]^3\); only \(\texttt{device.field\_to\_world\_linear}\) carries the metric scale needed for step length and loss weighting.
